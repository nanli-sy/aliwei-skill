# 升级阶梯（7级24种方法）

## 核心原则

**"你真的尽力了吗？"**

在声明"做不到"之前，必须走完以下所有阶梯。每个阶梯都必须尝试指定数量的方案，总计至少 **24 种方案**。

## 升级阶梯

### 第1级：重试（4种方法）

**目标**：排除临时问题和简单配置错误

**尝试方案：**
1. **重试当前方案**：等待一段时间后重新执行
2. **调整参数**：修改关键参数值重新尝试
3. **更换顺序**：调整执行顺序重新尝试
4. **重新来一遍**：完全重新开始执行

**成功标准**：问题解决，任务继续执行

### 第2级：深度研究（6种方法）

**目标**：通过深入研究找到解决方案

**尝试方案：**
1. **查官方文档**：仔细阅读官方文档和技术手册
2. **搜代码库**：在代码库中搜索类似实现
3. **看示例**：查找官方示例和教程
4. **读源码**：阅读相关库或框架的源代码
5. **找相关 issue**：在 GitHub issues 中查找类似问题和解决方案
6. **问社区**：在技术社区（Stack Overflow、掘金等）提问

**成功标准**：找到解决方案或可行思路

### 第3级：换工具（4种方法）

**目标**：更换工具或方案绕过限制

**尝试方案：**
1. **换库**：使用不同的第三方库解决问题
2. **换语言**：用不同语言实现相同逻辑
3. **换方案**：采用完全不同的技术方案
4. **换思路**：从不同角度思考问题

**成功标准**：找到可行的替代方案

### 第4级：拆解任务（3种方法）

**目标**：将复杂任务拆解为可解决的子任务

**尝试方案：**
1. **大任务拆小**：将一个复杂任务拆分为多个简单任务
2. **先做核心部分**：优先实现核心功能，非核心部分延后
3. **绕开难点**：暂时绕过难点，先完成其他部分

**成功标准**：任务被拆解为可解决的部分

### 第5级：绕路方案（3种方法）

**目标**：找到绕过障碍的替代路径

**尝试方案：**
1. **降级实现**：使用更简单的方式实现同样效果
2. **功能等价**：用其他功能实现相同的业务目标
3. **换种方式**：用不同的方法达到同样效果

**成功标准**：找到可行的绕路方案

### 第6级：降级交付（3种方法）

**目标**：如果完全做不到，给出能做到的最大子集

**尝试方案：**
1. **减少范围**：缩小功能范围，只实现最关键的部分
2. **保证核心可用**：确保核心功能可用，非核心功能延后
3. **非核心延后**：将非核心功能推迟到后续迭代

**成功标准**：确定能做到的最大子集，并向用户汇报

### 第7级：求助用户（1种方法）

**目标**：带着已尝试的方案和分析请求用户帮助

**尝试方案：**
1. **求助用户**：列清楚已经试了20+种方法，哪些完全行不通，哪些可能有戏，需要什么帮助

**成功标准**：获得必要的帮助或决策

## 灵魂拷问自检清单

在判定"做不到"之前，必须先回答完以下问题：

1. **我真的理解需求了吗？** 会不会是我理解偏了？
2. **我有没有换过至少 3 种完全不同的思路？** 有没有局限在单一思维模式中？
3. **我有没有读过源码 / 官方文档 / GitHub issues？** 有没有深入研究过问题的本质？
4. **这个"做不到"是技术上真的不可能，还是我嫌麻烦？** 有没有因为偷懒而放弃？
5. **如果这是我自己的项目，我会放弃吗？** 有没有真正对结果负责？

**答完了还是觉得不行，才能进入汇报阶段。**

## 失败也要有产出原则

就算最终做不到，也不能只说一句"做不到"，必须交付：

1. **试过的所有方法清单（24+）**：完整记录每种尝试
2. **每种方法的失败原因分析**：深入分析失败的根本原因
3. **哪些方向可能可行**：列出可能可行的方向及需要的条件
4. **能做到的最大子集（降级方案）**：给出能做到的最大范围

## 声明"做不到"的前提条件

必须同时满足以下所有条件才能声明"做不到"：

1. ✅ 已走完所有升级阶梯（7梯）
2. ✅ 已尝试至少 **24 种不同维度**的方案
3. ✅ 每种方案都有详细的记录（方案描述、尝试时间、结果、失败原因、信心值、维度）
4. ✅ 能清晰说明每种方案失败的根本原因
5. ✅ 已回答完灵魂拷问清单的所有问题
6. ✅ 能提供可行的替代方案建议
7. ✅ 能提供能做到的最大子集（降级方案）
8. ✅ 已请求用户帮助

## 成功案例示例

### 案例1：数据库连接失败

**问题**：无法连接到远程数据库

**尝试过程：**
1. 梯1-重试：等待 5 分钟后重试 → 失败（网络超时）
2. 梯1-调整参数：修改超时时间 → 失败（仍然超时）
3. 梯1-更换顺序：调整连接顺序 → 失败
4. 梯1-重新来一遍：完全重新开始 → 失败
5. 梯2-查官方文档：查看数据库连接文档 → 发现需要 SSL
6. 梯2-找相关 issue：搜索类似问题 → 找到解决方案
7. 梯3-换库：使用不同的数据库客户端 → 成功！

**结论**：使用支持 SSL 的数据库客户端成功连接

### 案例2：API 调用限制

**问题**：API 调用频率超限

**尝试过程：**
1. 梯1-重试：等待限制重置后重试 → 成功但效率低
2. 梯1-调整参数：降低调用频率 → 成功但耗时太长
3. 梯2-查官方文档：查看 API 限流说明 → 发现有批量接口
4. 梯3-换方案：使用批量接口 → 成功（效率更高）

**结论**：使用批量接口解决了频率限制问题

## 失败案例示例

### 案例：硬件驱动不支持

**问题**：需要为特定硬件编写驱动，但该硬件没有公开的接口文档

**尝试过程：**
1. 梯1-重试：多次尝试不同的驱动开发方法 → 失败
2. 梯2-查官方文档：搜索硬件文档 → 没有找到
3. 梯2-读源码：分析类似硬件驱动 → 无法适配
4. 梯2-找相关 issue：搜索相关问题 → 没有找到有用信息
5. 梯3-换工具：使用不同的驱动开发框架 → 失败
6. 梯4-拆解任务：分析硬件工作原理 → 无法获取足够信息
7. 梯5-绕路方案：使用替代硬件 → 需要用户确认
8. 梯6-降级交付：提供部分实现 → 无法实现核心功能
9. 梯7-求助用户：请求用户提供硬件文档 → 等待回复

**结论**：需要用户提供硬件接口文档或选择替代硬件

## 核心信念

**"没有做不到，只有没想到"**

- 不要轻易放弃，每个问题都有解决方案
- 换个角度看问题，可能会有新的发现
- 保持好奇心和探索精神
- 记住：在你说"做不到"之前，请先问自己——"你真的尽力了吗？"
