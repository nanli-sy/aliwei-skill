# 假设披露

## 核心原则

**所有假设必须明确说出来，不能藏着掖着。**

## 四类假设

### 1. 需求假设

关于用户需求的假设，即"我假设用户要的是X而不是Y"。

| 假设类型 | 示例 |
|----------|------|
| 功能理解 | "我假设你说的'快速排序'指的是原地排序算法" |
| 使用场景 | "我假设这个功能需要支持1000+并发用户" |
| 性能要求 | "我假设响应时间需要小于100ms" |
| 兼容性要求 | "我假设需要兼容IE11" |

### 2. 实现假设

关于技术实现的假设。

| 假设类型 | 示例 |
|----------|------|
| 技术选型 | "我假设使用React而不是Vue" |
| 架构设计 | "我假设采用微服务架构" |
| 依赖假设 | "我假设项目已安装了axios" |
| 环境假设 | "我假设运行环境是Node.js 18+" |

### 3. 质量假设

关于质量标准的假设。

| 假设类型 | 示例 |
|----------|------|
| 测试覆盖率 | "我假设单元测试覆盖率需要达到80%" |
| 代码规范 | "我假设遵循项目现有的ESLint配置" |
| 文档要求 | "我假设需要编写API文档" |

### 4. 范围假设

关于功能范围的假设。

| 假设类型 | 示例 |
|----------|------|
| 功能边界 | "我假设不包含移动端适配" |
| 不做的事情 | "我假设不修改数据库结构" |
| 优先级 | "我假设功能A优先于功能B" |

## 好假设 vs 坏假设

### 好假设的特征

| 特征 | 示例 |
|------|------|
| 具体明确 | "我假设需要支持IE11" |
| 有理由支撑 | "我假设需要支持IE11，因为用户提到有企业客户" |
| 可验证 | "我假设需要支持IE11，可以通过兼容性测试验证" |
| 影响范围小 | "我假设使用axios，项目已有此依赖" |

### 坏假设的特征

| 特征 | 示例 |
|------|------|
| 模糊不清 | "我假设需要支持旧浏览器" |
| 无理由支撑 | "我假设需要支持IE11" |
| 不可验证 | "我假设用户会喜欢这个设计" |
| 影响范围大 | "我假设需要重构整个架构" |

## 详细案例

### 案例1：需求假设披露

**用户需求：** "帮我优化一下首页性能"

**假设披露：**

```
🔴 关键假设披露：

1. 我假设你说的"优化性能"指的是页面加载速度，而不是运行时性能，因为你提到的是"首页"
2. 我假设需要达到的性能目标是：首屏加载时间 < 2秒，因为这是行业常见标准
3. 我假设不包含后端优化，只做前端优化，因为你没有提到后端
4. 我假设可以使用懒加载、代码分割等常见前端优化手段

🟡 约束清单：

- 不能引入新的依赖包
- 必须保持现有功能不变
- 优化后需要通过回归测试

🔵 歧义点确认：

1. 你说的"首页"是否包含登录页？
2. 是否有特定的性能指标需要达到？
3. 是否可以修改后端接口？
```

### 案例2：实现假设披露

**用户需求：** "帮我实现一个用户登录功能"

**假设披露：**

```
🔴 关键假设披露：

1. 我假设使用JWT进行身份验证，因为项目中已有jsonwebtoken依赖
2. 我假设密码需要加密存储，使用bcrypt算法
3. 我假设需要支持邮箱+密码登录，而不是手机号登录
4. 我假设登录成功后需要返回用户信息和token

🟡 约束清单：

- 必须使用项目现有的数据库连接
- 必须遵循项目的API规范
- 需要添加适当的错误处理

🔵 歧义点确认：

1. 是否需要支持第三方登录（如微信、GitHub）？
2. 是否需要记住登录状态？
3. 是否需要验证码？
```

## 假设验证流程

1. **披露假设**：在任务确认阶段列出所有关键假设
2. **用户确认**：用户确认假设是否正确
3. **执行验证**：执行过程中验证假设是否成立
4. **假设变更**：如果假设不成立，及时同步用户并调整方案
5. **记录结果**：交付时说明假设验证情况

**记住：** 任何未经验证的假设都是风险，必须披露并确认。
