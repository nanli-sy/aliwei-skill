# 质量门禁

## 核心原则

**交付前必须通过质量门禁，不允许"差不多就行"。**

## 四类检查项

### 1. 功能正确性检查

| 检查项 | 检查方法 | 通过标准 |
|--------|----------|----------|
| 核心功能是否正常 | 手动验证或运行测试 | 所有核心功能运行正常 |
| 边界情况是否处理 | 测试边界值和极端场景 | 边界情况有适当处理 |
| 错误处理是否完善 | 模拟异常输入 | 有合理的错误提示和处理 |
| 回归检查（改A坏B） | 检查相关功能 | 原有功能不受影响 |

### 2. 代码质量检查

| 检查项 | 检查方法 | 通过标准 |
|--------|----------|----------|
| 代码风格是否一致 | 对照项目规范 | 符合项目代码规范 |
| 是否有明显的代码缺陷 | 代码审查 | 无明显缺陷 |
| 命名是否清晰 | 人工检查 | 变量/函数命名有意义 |
| 注释是否足够 | 人工检查 | 复杂逻辑有注释说明 |

### 3. 安全性检查

| 检查项 | 检查方法 | 通过标准 |
|--------|----------|----------|
| 是否有敏感信息泄露 | 搜索敏感关键词 | 无敏感信息硬编码 |
| 是否有注入风险 | 代码审查 | 无SQL注入/XSS等风险 |
| 权限控制是否合理 | 功能测试 | 权限控制符合预期 |

### 4. 影响范围评估

| 检查项 | 检查方法 | 通过标准 |
|--------|----------|----------|
| 改动范围是否可控 | 查看变更文件 | 只改动必要的文件 |
| 是否有潜在的副作用 | 代码审查 | 无明显副作用 |
| 性能影响是否可接受 | 性能测试（如有） | 性能在可接受范围内 |

## 验证可信度评级

| 评级 | 定义 | 适用场景 |
|------|------|----------|
| **A级** | 自动化测试通过 | 有完善的测试覆盖 |
| **B级** | 手动验证通过 | 核心功能手动测试 |
| **C级** | 代码审查通过 | 无测试环境，仅代码审查 |
| **D级** | 未验证 | ❌ 不允许交付 |

## 假验证识别

以下情况视为"假验证"，不允许通过：

| 假验证类型 | 表现 | 正确做法 |
|------------|------|----------|
| 只跑了部分测试 | "测试通过了"，但只跑了一个测试用例 | 运行完整测试套件 |
| 没跑测试就说通过 | "应该没问题" | 必须实际运行测试 |
| 只验证了happy path | 正常流程通过，异常流程没测 | 必须测试异常场景 |
| 验证环境与生产不一致 | 开发环境通过，生产环境没验证 | 确认验证环境与生产一致 |

## 质量门禁通过标准

必须同时满足以下所有条件：

1. ✅ 功能正确性检查全部通过
2. ✅ 代码质量检查全部通过
3. ✅ 安全性检查全部通过
4. ✅ 影响范围评估全部通过
5. ✅ 验证可信度评级 ≥ B级
6. ✅ 无假验证行为

**未通过质量门禁的交付，视为不合格，必须退回重新修改。**
